# 11 — Internationalization (i18n)

## Motivation

Quill's UI is entirely in English. Supporting Chinese (Simplified) as a second language opens the app to a much larger audience and establishes the i18n foundation for future languages. Beyond UI strings, the AI reading assistant should also respond in the user's preferred language by default.

## Scope

- All user-facing UI strings externalized into translation files
- Language switcher in Settings (English / 简体中文)
- AI assistant system prompt adapted to respond in the selected language
- Suggested prompts, empty states, and toast messages translated
- Persist language preference in settings DB
- No changes to book content rendering (books are in whatever language they're written in)

## Implementation Approach

### Library: `react-i18next`

Lightweight, well-maintained, supports:
- JSON resource bundles
- Interpolation (`{count}`, `{title}`)
- Namespace support (optional, can use flat keys)
- React hook `useTranslation()` → `t("key")`
- Language detection and persistence

### Hook usage

```tsx
import { useTranslation } from "react-i18next";

function Sidebar() {
  const { t } = useTranslation();
  return <span>{t("sidebar.allBooks")}</span>;
}
```

### Interpolation

```tsx
t("reader.pageOf", { current: 1, total: 3 })
// → "Page 1 of 3" (en)
// → "第 1 页，共 3 页" (zh)
```

## AI Language Adaptation

### System prompt

The AI assistant's system prompt (in `src-tauri/src/commands/ai.rs`) is currently hardcoded in English:

```
"You are a helpful reading assistant. Help the user understand and discuss the book they are reading."
```

**Change:** Read the `language` setting from the DB. If `zh`, prepend a language instruction:

```
"You are a helpful reading assistant. Always respond in Chinese (Simplified). Help the user understand and discuss the book they are reading."
```

This is the simplest approach — no frontend changes needed for AI language adaptation. The system prompt controls the response language.

### Suggested prompts

The suggested prompt pills in AiPanel ("Summarize this chapter", etc.) are already externalized as `ai.prompt.*` keys. They will be translated so the user sees them in their language. The AI receives the translated prompt and responds accordingly.

### Auto-title

The `deriveTitle` function works on the raw user message text, which will be in the user's language. No changes needed — the title will naturally be in whatever language the user typed.

## Bilingual Dictionary

### Motivation

When a user reads a book in a foreign language, the current Look Up feature gives an English definition and contextual explanation — both in the book's language. But a Chinese-speaking user reading an English novel often just wants a quick native-language translation alongside the English definition. This is the most natural reading aid for non-native readers and a strong differentiator for Quill.

### How It Works

1. User sets a **mother language** in Settings (e.g., 简体中文)
2. When the user selects a word and triggers Look Up:
   - The app detects the book's language (from EPUB/PDF metadata, or inferred by the AI)
   - If the book's language differs from the user's mother language, the AI includes a **short native-language translation** at the top of the response
3. If the book is in the user's mother language, no translation is shown — behaves as today

### Response Format (Bilingual Mode)

```
┌─────────────────────────────────────────────┐
│  📖 Look Up                              ✕  │
│─────────────────────────────────────────────│
│                                             │
│  短暂的，转瞬即逝的                          │
│                                             │
│  **ephemeral** /ɪˈfɛm(ə)rəl/ · adjective  │
│  Lasting for a very short time; transient.  │
│                                             │
│  ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─  │
│                                             │
│  Here, the author describes the character's │
│  joy as ephemeral to foreshadow its loss —  │
│  the happiness is real but already passing. │
│                                             │
└─────────────────────────────────────────────┘
```

The native translation appears first — clean, short, no explanation. Just the meaning in the user's language. The rest of the response remains unchanged.

### Settings

Reuses the existing `language` setting as the mother language. No additional setting needed — if you set the app to Chinese, you're telling Quill "I think in Chinese."

A future enhancement could add a separate `mother_language` setting independent of UI language, but for v1, tying them together is the right default.

| Setting | Key | Values | Default |
|---------|-----|--------|---------|
| Language / Mother language | `language` | `en`, `zh` | `en` |

### System Prompt Change (`ai_lookup`)

When the book's language differs from the user's `language` setting, prepend an instruction to the existing lookup prompt:

```
Before the definition, provide a brief translation of the word/phrase in {mother_language}.
Keep the translation to a few words — no explanation, just the meaning.
Then proceed with the English definition and contextual explanation as usual.
```

### Book Language Detection

Three approaches, in order of preference:

1. **EPUB metadata** — the `<dc:language>` tag in EPUB OPF metadata (e.g., `en`, `zh`, `ja`). Most EPUBs have this. Store it in the `books` table as a `language` column.
2. **PDF metadata** — the `Lang` entry in the PDF document info dict. Less commonly set.
3. **AI inference** — if metadata is missing, let the AI infer the language from the passage text. This happens naturally — if the selected word is English and the user's language is Chinese, the AI will include the translation. No explicit detection needed.

For v1, rely on approach 3 (AI inference) to keep implementation simple. Add metadata-based detection later as a refinement.

### Implementation

This feature is implemented as part of Phase 4 (AI language adaptation):

1. In `ai_lookup` command, read the `language` setting
2. If `language != "en"`, append the bilingual instruction to the lookup system prompt
3. The AI handles the rest — no frontend changes to `LookupPopover` needed (it already renders streamed markdown)

### Edge Cases

- **User's language matches book language** — No translation line shown. The AI sees no bilingual instruction and responds as today.
- **Multi-word phrases / idioms** — Translation should capture the idiomatic meaning, not word-by-word. The prompt says "translate the word/phrase" which handles this.
- **Proper nouns** — Names don't need translation. The AI naturally handles this since the prompt says "a brief translation of the meaning."
- **Code / technical terms** — Some terms (API, JSON, etc.) are universal. The AI will naturally leave these untranslated or provide a brief functional description.

## Notes

- **No RTL support needed** — both English and Chinese are LTR
- **Font handling** — Inter supports basic Chinese but may need a fallback. Consider adding `"PingFang SC", "Microsoft YaHei"` to the font stack in `index.css`
- **Text length** — Chinese text is generally shorter than English, so layout overflow is unlikely. Verify anyway.
- **Adding languages later** — just add a new JSON file and a dropdown option. The architecture supports any number of languages.
